AMD Hybrid Inference Benchmark

COMPUTEX 2026 — NPU · GPU · Hybrid inference on AMD Ryzen AI. Auto-detects hardware, compares throughput, and shows AMD's three-stack AI strategy.

Benchmark Design

The script calls /v1/system-info on startup, detects available hardware, and automatically selects the matching benchmark scenario.

Hardware Detection

AMD XDNA 2 NPU (Ryzen AI 300+) → NPU-only vs GPU+NPU Hybrid 2-way comparison
AMD Radeon GPU / iGPU (Vulkan, no NPU) → CPU vs GPU benchmark
CPU only → CPU baseline only

WHY NPU vs HYBRID — NOT CPU or dGPU?

Both variants share the same ryzenai-llm recipe and model family, making TPS numbers directly comparable. Comparing against a discrete Radeon GPU (Vulkan GGUF) would use a different inference stack and different hardware — giving a misleading result. NPU vs Hybrid isolates the exact contribution of adding the iGPU to the NPU workload.

Usage

Run with auto-detected hardware or override any setting via command-line arguments.

# Auto-detect hardware and run the appropriate benchmark
python demo/computex_benchmark.py

# Override the GPU-only model
python demo/computex_benchmark.py --gpu-model Qwen3-1.7B-GGUF

# Skip model downloads (models already pulled)
python demo/computex_benchmark.py --skip-pull

# Skip backend installation
python demo/computex_benchmark.py --skip-install

# Connect to a non-default server
python demo/computex_benchmark.py --host localhost --port 13305

Learning

Deep dives into the three inference modes and AMD's AI software strategy.

Topic: ⚡ NPU Only 🎮 GPU Only 🚀 Hybrid ❓ Why No ROCm? 📊 AMD Stack 🔗 Integration 🏢 Vendors

NPU Only — Dedicated NPU Path

Example: Phi-4-mini-instruct-NPU

Runtime
AMD ryzenai-llm — runs entirely on Ryzen AI XDNA NPU. No GPU or CPU compute involved.
Power profile
Lowest power consumption — critical for laptops. Silent, low temperature, stable latency.
Throughput
Generally lower than Hybrid, but highly predictable latency characteristics.
Limitations
Model must be compiled to NPU format (INT8 / special compilation). Does not support large models due to NPU memory constraints.

GPU Only — GPU Inference Path

Example: Phi-4-mini-instruct-GGUF

Runtime
llama.cpp with Vulkan backend — cross-platform GPU inference, not AMD-specific.
Performance
Much faster than CPU. Highest model compatibility, most mature ecosystem with the largest community.
Portability
Vulkan backend works on AMD / NVIDIA / Intel GPUs across Windows and Linux.
Model format
GGUF — industry-standard quantized format. No compilation step required.

GPU + NPU — Hybrid Inference

Example: Phi-4-mini-instruct-Hybrid

Architecture
Some layers run on NPU, some on GPU (or CPU). The runtime automatically splits the workload.
Trade-off
Faster than NPU-only, more power-efficient than GPU-only. Best-of-both-worlds for AI PCs.
Runtime
ryzenai-llm — same recipe as NPU-only, enabling direct apples-to-apples TPS comparison.
GPU used
Integrated GPU (iGPU), not a discrete Radeon. Comparing against a dGPU would mix inference stacks and give a misleading result.

Hybrid inference is AMD's "best experience" answer for AI PCs: the NPU handles power-sensitive layers while the iGPU boosts throughput on compute-heavy layers — within the same ryzenai-llm stack.

Why "No ROCm"? — Key Insight

1 — ROCm ≠ Consumer Windows

ROCm is AMD's CUDA equivalent — primarily Linux-based, targeting datacenter (MI300) and AI training / HPC. Windows support is limited and unstable.

2 — llama.cpp chose Vulkan

Vulkan backend supports AMD / NVIDIA / Intel, runs on Windows and Linux, and requires no ROCm or CUDA. This portability path intentionally avoids ROCm.

3 — Ryzen AI stack ≠ ROCm

AMD's AI PC stack: NPU → ryzenai-llm, GPU → Vulkan / DirectML. ROCm is not part of this client stack. AMD does not push ROCm here.

4 — ROCm ≠ Edge / Client AI

Heavy installation, GPU compatibility constraints, driver–kernel coupling, and not suited for OEM or end-user distribution.

These three modes represent three product strategies: NPU → power efficiency & AI PC differentiation; GPU Vulkan → maximum compatibility; Hybrid → best experience. ROCm is intentionally excluded — it is not the optimal path for consumer AI PCs. This reveals AMD's strategic split: ROCm → datacenter (vs CUDA), Ryzen AI → client AI (a completely separate stack).

Vendor AI Stack Comparison

NVIDIA
CUDA — unified stack across consumer and datacenter. Single ecosystem, massive developer adoption and toolchain maturity.
AMD
Fragmented — ROCm (datacenter) + Ryzen AI (NPU) + Vulkan (consumer GPU). Three separate, incompatible stacks.
Intel
oneAPI — unified vision, but weak execution. NPU (AI Boost) + Arc GPU stack still maturing.
Apple
CoreML — closed but tightly integrated. Unified memory removes the NPU/GPU split entirely.

AMD's fragmentation is partly a consequence of its chip product line history (ATI GPU acquisition + independent XDNA NPU development). Whether this remains a structural constraint or becomes an opportunity for a unified client AI stack is AMD's defining challenge for the AI PC era.
